# コミットスキル

以下の手順でコミットを実行する。

## 手順

1. `git status` と `git diff --staged` を並列実行し、変更内容を把握する
2. ステージングされていない変更がある場合、ユーザーに対象ファイルを確認する
3. 変更内容を分析し、Conventional Commits形式でコミットメッセージを生成する

### コミットメッセージ形式

```
<type>: <簡潔な説明（日本語）>
```

#### typeの選択基準

| type | 用途 |
|------|------|
| feat | 新機能追加 |
| fix | バグ修正 |
| docs | ドキュメントのみの変更 |
| style | コードの意味に影響しない変更（フォーマット等） |
| refactor | バグ修正でも機能追加でもないコード変更 |
| test | テストの追加・修正 |
| chore | ビルドプロセスや補助ツールの変更 |

4. 変更内容に関連するオープンIssueがあるか確認する
   - `gh issue list --state open --limit 50 --json number,title,labels` で一覧を取得
   - 変更内容と関連しそうなIssueがあれば、クローズ対象としてユーザーに提示する
   - ユーザーが対象Issueを選択した場合、コミットメッセージ末尾に `Closes #N` を追記する（複数可）
5. 生成したコミットメッセージをユーザーに提示し、承認を得てからコミットする
6. コミット後、プッシュするか確認する
7. プッシュ実行後、`Closes #N` を含むコミットがmainブランチに入った場合、GitHubが自動でIssueをクローズする。自動クローズされない場合（ブランチ運用等）は `gh issue close N` で手動クローズする

## 注意事項

- 複数の論理的変更が混在する場合は、分割コミットを提案する
- .envや機密情報を含むファイルがステージングされていないか確認する
- Issueクローズは強制ではない。関連Issueがない場合やユーザーが不要と判断した場合はスキップする
